这份 wiki 从 Radarr 的进程入口开始，一层层拆到 Core 业务域、Provider 扩展体系、搜索下载流水线、API 与前端数据流。 每个大模块都配有模块组织图和运行流程图，便于顺着请求、命令、事件和数据模型定位代码。

Radarr 的数据访问不是 Entity Framework，而是 Dapper + 自定义表映射 + 仓储。主数据库和日志数据库在启动时初始化， migration 文件按编号演进 schema。

决策引擎

决策引擎把 release 变成下载候选。它先把标题映射到库中电影，再逐个执行规格检查，最后按质量、Custom Format、delay profile 等排序。

模块组织图 NzbDrone.Core/DecisionEngine
DownloadDecisionMaker主入口，生成 DownloadDecision。
Specifications质量、语言、大小、监控、历史、延迟等规则。
ParsingService把 ParsedMovieInfo 映射到 Movie。
Aggregation补充 seed、release profile、历史等上下文。
CustomFormats解析 release 命中的格式并计算分数。
DownloadDecisionComparer候选排序逻辑。
DownloadRejection拒绝原因和临时拒绝状态。
ProfilesQualityProfile、DelayProfile、ReleaseProfile。
运行流程图 decision evaluation
ReleaseInfo.Title例如电影名、质量、group。
→
Parser.ParseMovieTitle得到 ParsedMovieInfo。
→
ParsingService.Map匹配库中 Movie。
→
Specifications按 Priority 分组检查。
Custom Format Score用于排序和阈值检查。
→
DownloadDecision保存 accepted/rejections。
→
PrioritizeDecisions每部电影内按 comparer 排序。
规则类别 示例 specification 主要问题
电影匹配 Search/MovieSpecification 这个 release 是否属于当前搜索电影。
质量和语言 QualityAllowedByProfile, LanguageSpecification 候选是否符合 Quality Profile。
下载安全 Blocklist, Queue, FreeSpace 是否已下载、被屏蔽、队列已有、磁盘是否够用。
RSS 专属 Delay, MonitoredMovie, Availability 自动抓取时是否需要等待，电影是否监控，是否已发布。
偏好排序 DownloadDecisionComparer 候选之间谁更好，不一定等同于谁可下载。

Custom Filter 体系

Custom Filter 是前端列表筛选能力。后端保存 filter 定义，前端根据列表类型取出 filter 并在 selector 中执行。 本地 R005 重点扩展了嵌套 AND/OR 条件组。

模块组织图 R005 nested filter
CustomFilterController保存和读取自定义筛选器。
FilterBuilder编辑条件行和条件组。
filterTree utilitiesnormalize、isFilterGroup、combinator。
createClientSideCollectionSelector递归执行条件树。
Legacy flat filters简单 AND 仍保持兼容。
Nested groups支持 group 内 AND/OR 和子 group。
Server-side guard只有 flat AND 可安全下推到后端查询。
运行流程图 filter execution
User selects filter例如交互搜索或队列表格。
→
findSelectedFilters找到内置或自定义 filter。
→
normalizeFilterGroup把旧数组转成根 AND 组。
→
doesFilterGroupMatch递归执行 AND/OR。
R005 的关键原则是兼容旧数据：简单 filter 继续序列化为 flat AND；只有用户使用条件组时，前端才按树结构执行。

Prowlarr 媒体信息接入专题

本地 R001 把 Prowlarr 返回的音轨语言、字幕语言、每语言音频规格和 MediaInfo enrich 状态接入 Radarr。 这条链路横跨 Newznab/Torznab parser、ReleaseResource、交互式搜索 UI 和自动搜索等待逻辑。

模块组织图 R001 downstream media fields
Torznab/Newznab attrsaudio, subs, mediaInfo progress。
ReleaseInfoSubs, AudioInfo, MediaInfoStatus。
ReleaseResource/api/v3/release JSON 输出。
InteractiveSearchAudio Info、Subtitle Languages、progress alert。
/release/mediaInfoRadarr 代理到 Prowlarr 的 per-row enrich。
releaseActions.js前端队列请求附加数据并合并行状态。
Automatic Search Wait非交互搜索等待 pending mediaInfo 再决策。
运行流程图 Prowlarr mediaInfo enrichment
Prowlarr search result先返回基础 release 和 pending 状态。
→
Radarr Torznab parser解析 audio/subs/progress attrs。
→
Interactive Search UI显示行级 loading 和整组进度。
→
POST /release/mediaInfo触发 Prowlarr 单行 enrich。
Prowlarr response返回该行 audio/subs/status/progress。
→
Radarr cache update更新 remoteMovie cache。
→
UI merge更新完整 release list，不主动改变当前排序快照。
字段 后端承载 前端用途
audioInfo ReleaseInfo.AudioInfo 和 ReleaseResource.AudioInfo 显示语言和规格，例如 English: TrueHD Atmos、Chinese: DDP 5.1。
subs ReleaseInfo.Subs 和 ReleaseResource.Subs 显示字幕语言，可参与搜索结果筛选和排序。
mediaInfoStatus pending, available, failed, unavailable 控制单行 loading 是否继续显示。
mediaInfoProgress* 整组搜索进度，completed/total/status。 控制交互式搜索顶部的 additional data 进度提示。
prowlarrIndexerId Radarr indexer id 和 Prowlarr 内部 indexer id 的桥。 让 Radarr 能调用 Prowlarr 的 per-row mediaInfo 端点。

开发者索引

这一节不是模块介绍，而是日常改代码时的入口地图。顺着这些路径能快速从需求定位到应该修改的层。

追踪一个 API 请求

从 Radarr.Api.V3/*Controller 找 controller，再看 resource mapper，最后进 Core service。

新增 Release 字段

改 ReleaseInfo、RSS parser、ReleaseResource、前端 typings/actions/table/filter。

新增 Indexer

实现 IIndexer 或继承 HttpIndexerBase，补 request generator、parser、settings、测试。

新增下载客户端

看 Download/Clients 同类实现，接入 provider schema、连接测试和 Download 方法。

新增前端表格列

改页面 columns、row component、store sort/filter predicates、translation。

新增 Custom Filter 字段

为对应 collection 补 filterBuilderProps、filterPredicates，必要时补 API resource 字段。
